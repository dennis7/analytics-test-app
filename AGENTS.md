# AGENTS.md

## Project Overview

ESG analytics pipeline monorepo with two parallel implementations (dbt and SQLMesh) producing identical outputs, plus a generic MCP server for DuckDB exploration. All three packages share a single uv workspace.

## Commands

All commands run from the repo root via `just`. Never use raw `uv run dbt` or `uv run sqlmesh` directly -- always use the justfile recipes which set `DATA_DIR` and other required flags.

```bash
just sync              # Install deps (Python + dbt packages)
just seed              # Generate parquet seed data
just lint              # Ruff check + format
just dbt build         # Run full dbt pipeline
just dbt test          # Run dbt tests only
just sqlmesh plan      # Run SQLMesh pipeline
just sqlmesh audit     # Run SQLMesh audits only
just test-mcp          # Run MCP integration tests (pytest)
just mcp               # Start MCP server
```

### Single-date filtering

Both pipelines read the `REPORTING_DATE` env var. When unset, all dates are processed.

```bash
REPORTING_DATE=2025-12-31 just dbt build
REPORTING_DATE=2025-12-31 just sqlmesh plan
```

## Architecture

### Workspace layout

```
packages/
  dbt/         # dbt-duckdb implementation
  sqlmesh/     # SQLMesh implementation
  mcp/         # MCP server (generic DuckDB explorer)
scripts/
  generate_seed_data.py   # Seed data generator
data/          # gitignored -- parquet inputs + DuckDB outputs
```

### Pipeline structure (identical in both dbt and SQLMesh)

Each metric (WACI, WAGR) follows the same 4-layer pattern with dedicated schemas:

```
staging (shared)  ->  preprocess  ->  harmonize  ->  waterfall  ->  metrics
10 stg_* views        2 models        2 models       2 models      1 model
```

- **Staging**: Type casting from parquet, optional date filtering. Shared across metrics.
- **Preprocess**: Join positions with market data + security master. Per-metric (`waci_prep_*`, `wagr_prep_*`).
- **Harmonize**: Compute market values and portfolio weights. Per-metric.
- **Waterfall**: Metric-specific calculations (carbon intensity lookup / green revenue lookup, then weighted values).
- **Metrics**: Final aggregation per portfolio per date.

### Shared logic

- **dbt**: Jinja macros in `packages/dbt/macros/shared/` (called from per-metric models)
- **SQLMesh**: Python `@macro()` functions in `packages/sqlmesh/macros/shared.py` that load SQL templates from `.sql` files in the same directory

### Data quality

- **dbt**: 32 singular SQL tests in `packages/dbt/tests/` (organized by layer: staging/, waci/, wagr/)
- **SQLMesh**: 35 audits in `packages/sqlmesh/audits/` (same structure), attached to models via `audits (...)` in MODEL blocks

### MCP server

Generic DuckDB explorer -- no dbt/sqlmesh dependency. Accepts 1-2 `--db` paths, ATTACHes them read-only. Three tools: `list_tables`, `describe_table`, `query`. 20 integration tests.

## Conventions

### General

- Python 3.12+, managed exclusively by **uv** -- never use pip, pip install, or pip freeze
- All build, test, and run commands go through **just** -- never invoke uv run, dbt, sqlmesh, or pytest directly
- Ruff for linting and formatting (line-length 120)
- Run `just lint` before committing
- All tests must pass before committing: `just dbt build`, `just sqlmesh plan`, `just test-mcp`

### SQL models

- dbt and SQLMesh implementations must stay in sync -- same model names, same logic, same outputs
- Staging models are views; everything else is materialized as tables
- Model naming: `{metric}_{layer}_{description}` (e.g., `waci_prep_public_securities`, `wagr_wf_green_revenue`)
- Every model must have at least one data quality test/audit

### Testing

- dbt tests live in `packages/dbt/tests/{layer}/`
- SQLMesh audits live in `packages/sqlmesh/audits/{layer}/`
- Test naming: `assert_{model_short_name}_{check}` (e.g., `assert_stg_entity_carbon_not_null`)
- MCP tests: `packages/mcp/tests/` using pytest + anyio

### Seed data

- Generated by `scripts/generate_seed_data.py`, writes to `data/{env}/input/`
- Two environments: dev and test
- Each environment has 2 month-end dates
- Data files are parquet, gitignored

### Logging

- All logs go to `logs/` (gitignored)
- dbt: rotated per run (timestamped files)
- SQLMesh: rotated per run (timestamped files, built-in)
- MCP: RotatingFileHandler (5MB max, 3 backups)

### Git

- Don't commit generated data, logs, or DuckDB files
- Don't commit `.env` files
- `packages/dbt/.user.yml` is gitignored
